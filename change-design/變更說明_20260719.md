## 第二十二批新增內容（本次）：修正跨分支圈選建甘特時的 JS 錯誤

### 問題
跨分支圈選建甘特時，Console 報錯：
`Cannot read properties of undefined (reading 'method') at showGanttSegmentPicker`

### 原因：潛伏已久的 bug 被跨分支功能第一次引爆
甘特的「段落選擇器」（框到多個段落時跳出的視窗）讀取 `e.segment.method`，
但建立資料的程式**從來沒有提供過 `segment` 這個欄位**——連上一批修改之前的
原始版本也沒有。以前單分支框選只會產生 1 筆結果、走不到這個多筆選擇器，
這顆雷潛伏至今沒人踩過；跨分支圈選會常態產生 2 筆以上，第一次真正執行到
這段程式碼就爆了。

### 修法
選擇器改用資料裡實際存在的欄位顯示（段落編號、節點區間、小段數），
並把節點名稱一併傳給甘特面板（它本來就支援，這樣甘特項目標籤會顯示
「節點B至交會點」而不是編號）。

### 涉及檔案
- **`js/batch.js`**：修正 `showGanttSegmentPicker`。
- **`index.html`**：版本號推進到 `?v=20260718`。

### 部署後驗收
1. 跨分支框選（主線節點＋支線節點）→ 圈選建甘特 → 選擇器正常跳出、

## 第二十一批新增內容（本次）：圈選支援跨分支（支線節點＋主線節點）

### 需求
框住「支線的節點A＋主線的節點B」（各分支只有 1 個節點）時，
三個圈選功能原本都會忽略單節點的分支；你希望這樣也能建甘特/更新完工/設定屬性。

### 修法：自動計算「交會點」補齊範圍
三個圈選功能改用同一套跨分支範圍計算：
- 同一分支框到 2 個以上節點 → 取最小~最大（原本行為，完全不變）。
- **只框到 1 個節點的分支 → 自動延伸到與另一個被選分支的「交會點」**：
  框住支線節點A＋主線節點B，會得到兩個範圍——
  主線【節點B ~ 交會點】＋支線【交會點 ~ 節點A】，正好涵蓋兩點之間的完整管路。
- 交會點以幾何計算（點到線段投影，容差 1m），Y 型分岔與頭尾串接兩種結構都支援；
  單節點分支若與其他被選分支根本不相接，維持忽略並提示。

### 三個功能的承接方式
- **圈選建甘特**：兩個範圍各建一筆甘特項目（面板本來就支援多筆預選）。
- **圈選更新完工**：跳出段落選擇器，兩個範圍逐一設定（各選一次）。
- **圈選設定屬性**：對話框**一次列出全部範圍**，管徑/工法填一次、
  跨分支同時套用（對話框已改為多範圍版本）。

### 涉及檔案
- **`js/batch.js`**：新增共用的 `computeRectNodeRanges()`（含交會點拓撲），
  三個圈選 finish 全部改接。
- **`js/map.js`**：範圍屬性對話框支援多範圍一次套用。
- **`index.html`**：版本號推進到 `?v=20260717`。

### 部署後驗收
1. 框住「主線節點B＋支線節點A」→ 圈選設定屬性 → 對話框應列出兩個範圍
   （主線 節點B→交會點、支線 交會點→節點A），填一次套用到全部。
2. 同樣的框選 → 圈選建甘特 → 應產生兩筆預選段落。
3. 同樣的框選 → 圈選更新完工 → 段落選擇器列出兩個範圍，各自可設定。
4. 原本同分支 2 節點的用法 → 行為完全不變。

### 已知限制
「圈選更新完工」跨分支時是逐段設定（選擇器選一個設一個）；若希望改成
「一次套用到全部段落」，跟我說一聲，下一批補上。

# 第十八批內容（前次交付）：照片跟著管段走

---

## 第十八批新增內容（本次）：編輯路徑後，小段照片跟著實際管段走

### 起因
你問「編輯路徑之後，小段內的照片會怎麼處理」。查了程式碼，誠實回答：
**原本的處理有明顯缺陷，其中一個會直接刪照片**：
1. **段落變短**：超出新長度的小段上的照片，資料庫紀錄**直接被刪除**——
   照片從系統中消失（R2 實體檔變孤兒）。
2. **中段增減長度**：第八批讓「屬性」跟著實際位置走了，但照片還是死守編號，
   編輯後照片會掛到錯誤的物理位置。
3. **最危險——分界移動**（第十三批）：把節點2 往回拖 40m 時前段縮短，
   掛在被劃走那 40m 上的照片會觸發第 1 點**被刪掉**，而不是跟著管段搬去後段。

### 修法：照片比照屬性，跟著實際位置走，且永不因編輯被刪
- 編輯路徑存檔時，照片依「舊位置 → 新位置」的對應表自動搬家，
  含跨段的情況（分界移動時，被劃走管段上的照片跟著搬去另一段）。
- 該區域被壓縮、找不到精確對應時，搬到來源編號最接近的新小段（不會憑空消失）。
- 即使是不帶對應表的舊式編輯，超出範圍的照片也只會**夾回該段最後一個小段**，
  **一律不再刪除**。

### 模擬驗證（分界移動情境：B0 190m→150m、B1 280m→320m）
- 照片在不受影響區（B0 第 5 段）→ 原地不動 ✓
- 照片在被劃走的 40m 上（B0 第 17 段）→ **跟著管段搬到 B1 第 2 段**
  （舊版這張會被刪除！）✓
- 照片在後段（B1 第 10 段）→ 編號平移到 B1 第 14 段，物理位置不變 ✓

### 涉及檔案
- **`pipe-map-worker/src/index.js`**：`updateLinestring` 移除刪照片邏輯，
  加入照片重掛。**需要重新部署。**
- 前端無需變動（對應表本來就會送）。

### 部署後驗收
1. 找一個有照片的小段，記下照片內容與位置。
2. 編輯路徑，讓該小段前方的長度改變（縮短或加長 10~20m）→ 存檔。
3. 照片應該還在**同一個物理位置**的小段上（跟著管段走了），沒有消失。
4. 做一次分界移動（拖分界節點），讓有照片的管段被劃到另一區間 →
   照片應跟著出現在另一區間的對應小段上。

### 提醒
這個保護**只對部署之後的編輯有效**——過去因編輯而被刪除的照片紀錄無法復原
（R2 裡的實體檔案還在，若有需要救回特定照片，可以再跟我說，寫個比對腳本
從 R2 找回檔案重新掛上）。

## 第十六批新增內容（本次）：節點框與段落標籤自動錯開，不再疊在一起

### 問題
兩條管線平行靠得很近（或節點密集）時，「節點X」名稱框跟管線上的
「工法 完工/總長」文字標籤全部疊在同一個位置，卡成一團完全讀不到。

### 修法：標籤防碰撞（collision avoidance）
繪製完成與每次縮放後，自動掃一遍畫面上所有可見的文字標籤：
- 發現兩個標籤重疊 → 把後面的往下推開，**垂直排成一疊**，每個都保持可讀。

## 第十批新增內容（本次）：修正「明明沒在編輯路徑，卻被擋下拖曳」

### 問題
上一批加的「拖曳節點名稱」功能，你回報即使沒有在編輯路徑，拖曳時還是跳出
「請先結束路徑編輯，再拖曳節點名稱」。

### 原因：一個被放大的既有舊 bug
進入「編輯路徑」模式時，程式碼只清除這一種圖層：
```js
if (layer instanceof L.Polyline) { map.removeLayer(layer); }
```
但節點名稱標記是用 `L.Marker` 畫的、制水閥中點是 `L.CircleMarker`——
**兩者都不是 `L.Polyline`，這個清除邏輯從一開始就沒把它們清掉**。
只是以前節點名稱標記不能拖曳，沒人會發現它還殘留在畫面上；這次幫它加上拖曳功能後，
這個殘留標記就變成一個隱形陷阱：進入編輯模式時，舊的節點名稱標記其實還疊在
畫面上（很可能疊在編輯用的白色幾何節點附近）。你以為在拖白色圓圈，
實際上抓到的是底下殘留的節點標記，於是正確觸發了警告——但這對你來說是意外行為。

簡單說：這是一個從很久以前就存在、但因為「以前東西不能拖，所以沒人踩到」的
潛在問題，這次因為新功能而第一次被實際踩到。

### 修法
進入編輯模式時，不管圖層型別，**全部清乾淨**（節點名稱標記、制水閥標記、
管線色帶通通清掉）。退出編輯模式時本來就會重新整頁繪製，這些標記屆時會
依最新資料重新畫出來，不影響資料本身。

### 涉及檔案
- **`js/path-edit.js`**：`startEditMode()` 的圖層清除邏輯改為清除全部圖層型別。
- **`index.html`**：版本號推進到 `?v=20260707`。

### 部署後驗收
1. 確認目前不在編輯模式（沒看到「編輯中...」的按鈕狀態、沒有編輯工具列）。
2. 拖曳節點名稱標記 → 應該正常跳出搬移確認視窗，不再出現「請先結束路徑編輯」。
3. 進入編輯路徑模式一次，確認畫面上乾淨（沒有殘留的舊節點名稱標記、制水閥紅十字
   疊在編輯用的白色節點旁邊），只看到編輯用的橙色/紫色管線與白色幾何節點。
4. 退出編輯模式（不論存檔或取消）→ 節點名稱標記、制水閥應該正常重新出現。

# 第九批內容（前次交付）：節點名稱可用滑鼠拖曳搬移

---

## 第九批新增內容（本次）：「節點1/節點2」可以直接用滑鼠拖曳搬移

### 先解釋一個重要觀念：地圖上有兩種「節點」
1. **幾何節點**（路徑編輯模式裡的白色圓圈）：管線線形的轉折點，
   進「編輯路徑」後本來就可以拖曳，拖它會改變管線的形狀和長度。
2. **命名節點**（「節點1」「節點2」這種有名字的標記）：它不是線形的一部分，
   而是掛在某個小段上的**屬性**（節點名稱），畫在該小段的起點位置。
   你截圖裡節點2 的小圓點跟白色大圓圈不重疊，就是這個原因。

你問的「編輯路徑時能不能移動節點2」——白色圓圈本來就能拖（那是改線形）；
但**命名節點原本沒有任何拖曳功能**，要搬它得用很迂迴的方式：
右鍵舊小段→清空名稱→右鍵新小段→重打名稱。這批直接做成可以拖曳。

### 新功能：拖曳搬移命名節點
- 在**一般檢視模式**（非路徑編輯）下，直接用滑鼠按住節點的小圓點拖曳。
- 放開時自動**吸附到最近的小段**，跳出確認視窗（顯示目標分支與距離），
  確認後一步完成搬移（後端把節點名稱從舊小段移到新小段），地圖即時重繪，
  段落標籤上的「節點A → 節點B」文字也會跟著更新。
- 防呆機制：
  - 放開位置離管線超過 100m → 取消並彈回原位。
  - 目標小段已有別的節點名稱 → 提示先清除，彈回原位。
  - 未登入 → 彈回原位（沿用既有權限機制）。
  - 正在路徑編輯模式 → 提示先結束編輯（避免跟幾何節點拖曳混淆）。
  - 放回原本的小段、或按取消 → 無事發生，彈回原位。
- 後端不用改：既有的 `updateSmallSegmentInfo` 本來就支援只更新節點名稱。

### 附帶修好另一個既有 bug：單一 LINESTRING 管線的節點名稱標記從來不會顯示
跟先前修過的「制水閥畫不出來」是**同一類遺漏**：畫節點名稱標記的程式碼
只存在於「有分支」的管線（MULTILINESTRING）的繪製路徑裡，
單純一條線的管線（LINESTRING）就算在小段上設定了節點名稱，地圖上也看不到。
這次把節點標記抽成共用函式（順便掛上拖曳功能），兩種管線類型都會畫、行為一致。

### 涉及檔案
- **`js/map.js`**：節點標記抽成共用函式 `drawNodeNameMarker`（含拖曳搬移邏輯
  `handleNodeMarkerDrag`），MULTI 與單一 LINESTRING 兩個繪製路徑都改用它。
- **`index.html`**：版本號推進到 `?v=20260706`。

### 部署後驗收
1. 放大到能看到節點標記的縮放層級，按住「節點2」的小圓點拖到別的位置放開 →
   跳出確認視窗 → 確認 → 節點標記移到新位置，段落標籤的節點區間文字同步更新。
2. 拖到離管線很遠的地方放開 → 提示「離管線太遠」並彈回原位。
3. 拖到已經有其他節點名稱的小段上 → 提示先清除，彈回原位。
4. 未登入狀態下拖曳 → 彈回原位（不會誤改資料）。
5. 找一條**單一線形（無分支）**、小段上有設定節點名稱的管線 →
   節點標記現在應該會顯示了（之前完全看不到），而且一樣可以拖曳。

# 第八批內容（前次交付）：路徑編輯不再「從尾巴扣」

---

## 第八批新增內容（本次）：編輯路徑後，小段屬性跟著「實際位置」保留

### 問題（你回報的）
編輯路徑微調節點1~節點2 之間的線形（250m → 240m）之後：
節點1~節點2 的標示仍是 250m，少掉的 10m 卻出現在最尾端（節點3 那邊被吃掉一截）。
每次微調中間，整條管線後半的資料就整個往前滑，非常麻煩。

### 原因
小段資料（每 10m 一段的管徑/工法/狀態/節點名稱）是用**編號**存的
（第 0 段=0~10m、第 1 段=10~20m…）。存檔時後端重新產生小段，
「新的第 N 段繼承舊的第 N 段」——**按編號硬對齊，不管實際位置**。
你把中段縮短 10m 後，後面所有屬性的編號沒變，但整條線短了 10m，
於是所有東西物理上往前滑 10m，一路推到底，最後 10m 的資料就消失了。

### 修法：屬性改成「跟著實際位置走」
- **前端（`js/path-edit.js`）**：存檔時比對新舊路徑，把「沒被你移動的節點」
  （新舊座標誤差 <5cm）當作**錨點**，在錨點之間建立新舊距離的線性對應，
  算出一張「新小段編號 → 舊小段編號」的重對應表，隨存檔請求一起送出。
  效果：你編輯的那段自己吸收長度變化，錨點之後的所有屬性照實際位置原位保留。
- **後端（worker `updateLinestring`）**：收到重對應表時改用它繼承屬性；
  節點名稱另外特別處理——路徑拉長時同一個舊小段可能對應多個新小段，
  直接繼承會出現重複的節點標記，所以一個舊節點名稱只落在一個新小段上。
  沒收到表（例如舊版前端）就退回原本行為，完全向下相容。

### 驗證（端到端模擬，模擬你回報的精確情境）
建一條 593m 的線（節點1→節點2 推進 250m、節點2→節點3 埋設 300m、水管橋 43m），
把節點1~節點2 之間的繞路拉直（縮短 10m），跑完整的重對應流程後確認：
- 工法分界正確落在 **240m**（推進→埋設）與 **540m**（埋設→水管橋）✓
- 被編輯的前段自己變成 24 個小段（240m），**自己吸收了縮短量** ✓
- 節點2、節點3 名稱標記跟著實際位置移到 240m / 540m ✓
- 尾端水管橋 43m 完好無損，**不再被吃掉** ✓

### 涉及檔案
- **`js/path-edit.js`**：存檔時計算重對應表（新增 3 個輔助函式）。
- **`pipe-map-worker/src/index.js`**：`updateLinestring` 支援 `indexRemap` 參數。
- **`index.html`**：版本號推進到 `?v=20260705`。

### 部署後驗收
1. 找一條有多段工法、有節點名稱的管線，編輯路徑：把中間某段稍微拉直或繞遠一點
   （改變該段長度），存檔。
2. 確認：被你調整的那段長度改變（例如 250→240），**它後面的每一段長度維持不變**、
   節點名稱標記跟著移動到正確位置、最尾端的資料完好。
3. 也測一下「把某段拉長」的情況（例如多繞 20m）→ 後面資料一樣原位保留，
   被拉長的區域內小段屬性以原屬性延伸填滿。
4. 提醒：這個修正**只對「這次部署之後」的編輯有效**，之前已經位移掉的資料
   沒辦法自動還原，需要手動把錯位的段落改回來（改一次之後就不會再跑掉了）。

# 第七批內容（前次交付）：修正切回大地圖制水閥消失

---

## 第七批新增內容（本次）：修正「進小地圖再跳回大地圖，制水閥消失」

### 問題
第一次進大地圖能看到制水閥；點進個別工程（小地圖）看一下，再切回大地圖 →
制水閥不見了。

### 原因
制水閥是在「背景載入各工程小段資料」那個迴圈裡順便畫出來的
（`_loadProjectProgressBackground`），但那個迴圈**只處理「還沒載入過」的工程**
（用 `pipeline._progressLoaded` 這個旗標判斷）。

第一次進大地圖時，工程資料都是新載入的，迴圈會跑、制水閥會畫出來。
點進小地圖時 `clearMap()` 會把地圖上所有圖層（含制水閥標記）清空——但
**工程的小段資料本身還留在記憶體裡**，`_progressLoaded` 依然是 `true`。
切回大地圖時，這些工程對背景載入迴圈來說「已經載入過了」，直接被跳過、
完全不會執行到，制水閥自然沒機會被重新畫出來。

簡單說：資料還在，只是「畫出來」這個動作只在「第一次載入資料」的當下觸發過一次，
之後圖層被清空就沒人記得要重畫了。

### 修法
在 `showProjectPipelines()`（大地圖總覽的進入點）多加一步：對「資料已經在、
只是圖層剛被清空」的工程（也就是 `pipeline.branches` 已經有值的），
**進入畫面時立刻補畫一次制水閥**，不用等背景載入迴圈（反正那個迴圈本來就不會
再處理這些工程）。還沒載入過的工程則完全不受影響，照舊交給背景載入處理。

這兩種情況（已載入 vs 未載入）互斥，不會有同一個工程的制水閥被畫兩次、重疊變粗的問題。

### 涉及檔案
- **`js/data.js`**：`showProjectPipelines()` 內新增前述的補畫邏輯。

### 部署後驗收
1. 進入計畫大地圖 → 確認制水閥（紅色垂直線）有顯示。
2. 點進任一個工程（小地圖）看一下。
3. 點「返回」切回大地圖 → 制水閥應該**還在**（這是這次修的重點）。
4. 重複步驟 2-3 幾次（換不同工程進出）→ 制水閥應該每次都還在，不會累積變多、
   也不會消失。

# 第五批內容（前次交付）：修正登入時 `ERR_CONNECTION_CLOSED`

---

## 第五批新增內容（本次）：修正登入時 `ERR_CONNECTION_CLOSED`

### 問題
你回報登入時卡在「驗證權限中」，console 顯示：
```
verifyUser&...&token=eyJhbGci...  Failed to load resource: net::ERR_CONNECTION_CLOSED
驗證失敗: TypeError: Failed to fetch
```
先排除了「Worker 整個掛掉」的可能——你測試 `getProjects` 完全正常，證實 Worker 本身健康，
問題只出在 `verifyUser` 這一個特定請求。

### 原因
`login.html`（獨立的登入頁面，跟主系統 `index.html` 內建的「重新登入」彈窗**不是同一份程式碼**）
把整個 Google ID Token 用 GET 塞進 URL query string。實際算了一下你附的那個失敗請求，
**URL 長度高達 1407 字元**，裡面整段是看起來像「憑證/token」的 base64 字串。這種內容
很容易被某些網路環境（防火牆、防毒軟體、行動網路的內容過濾器）判定為可疑，直接把連線
砍斷——這正好對應到 `ERR_CONNECTION_CLOSED`（連線被斷開，不是收到一個錯誤回應）。

順帶一提：主系統內建的「重新登入」彈窗（`js/auth.js`）**本來就是用 POST**，
所以那條路徑沒有這個問題；只有獨立登入頁 `login.html` 這條路徑是用 GET。

### 修法
把 `login.html` 的 `verifyUser` 請求從 GET + URL query 改成 POST + body
（`application/x-www-form-urlencoded`，Worker 對 GET query 和 POST body 用同一套
解析邏輯，後端完全不用改）。這也是比較正確的作法——Google ID Token 是敏感憑證，
本來就不該整包放在 URL 上（會進瀏覽器歷史、任何中間節點的存取 log）。

用 `application/x-www-form-urlencoded` 而不是 `application/json` 是刻意的：
前者屬於瀏覽器認定的「簡單請求」，不會觸發 CORS 預檢（OPTIONS），
維持原本程式碼註解裡「避免 CORS preflight」的目的，只是換一種方式達成。

### 涉及檔案
- **`login.html`**：`verifyAndRedirect()` 改為 POST body 傳送。

### 部署後驗收
1. 用無痕視窗開啟登入頁面，走一次完整登入流程 → 應該能正常驗證成功並跳轉，
   不再卡在「驗證權限中」。
2. 開 DevTools Network，確認 `verifyUser` 這筆請求方法是 `POST`，
   網址列只剩 `?action=verifyUser`，token 不再出現在 URL 上。
3. 如果**還是**斷線：代表原因不是 URL 長度（可能是更底層的網路環境問題），
   麻煩告訴我，我們再往別的方向查（例如換一個網路環境測試、檢查瀏覽器擴充功能）。

## 第三批新增內容（本次）：大地圖顯示制水閥

### 需求
個別工程地圖（點進單一工程）原本就會在制水閥位置畫紅色十字標記；
但切回「計畫總覽」大地圖（顯示整個計畫所有工程，如你附圖 2 那種多色路線總覽）時，
制水閥完全不會顯示。這次讓大地圖也自動顯示。

### 運作方式
制水閥的位置本來就不是存在資料庫裡的經緯度，而是「這條管線的座標 + 小段的
起訖距離」即時內插算出來的——跟個別工程地圖用的是同一套算法，**不需要新增
任何後端 API 或資料庫欄位**。大地圖背景載入各工程進度時（`_loadProjectProgressBackground`，
本來就會為了算統計數字去抓每個工程的小段資料），現在額外多做一步：
掃過該工程所有標記 `isValve` 的小段，算出中點座標，畫成標記。
因為是背景載入，你會看到制水閥標記隨著工程一個一個載入陸續浮現在大地圖上。

### 標記樣式的選擇
個別工程地圖那種「紅色十字線」是以公尺為單位換算的固定地理長度（約 40 公尺），
放到動輒橫跨數公里的計畫總覽地圖上會小到幾乎看不見。大地圖改用
Leaflet 的 `circleMarker`（固定像素半徑），**不管地圖縮放到多遠，畫面上永遠是清楚的
一個小紅點**，滑鼠移上去會顯示提示「🔧 制水閥」，點下去會彈出小視窗顯示所屬工程名稱。

### 附帶修好一個既有 bug：單一 LINESTRING 管線的制水閥原本畫不出來
檢查程式碼時發現：個別工程地圖裡，只有「有分支」的管線（MULTILINESTRING）
在畫小段時才會呼叫畫十字線的函式；**沒有分支、單純一條線的管線（LINESTRING）
完全沒有呼叫這個函式**，所以就算標記了制水閥也不會顯示紅十字。
這次一併補上，兩種管線類型現在行為一致。

- **`js/plan-overview.js`**：
  - 新增 `drawProjectValveMarkers(pipeline)` 主函式。
  - `clearMap()` 一併清除 `projectValveMarkers`（切換工程/返回大地圖時不會殘留舊標記）。
  - `_loadProjectProgressBackground()` 載入完某工程的小段資料後，若使用者仍停留在
    「同一個計畫」的大地圖（沒切走、沒點進別的工程），才畫上該工程的制水閥——
    避免非同步載入完成時使用者已經切到別處，畫錯地方。
- **`js/map.js`**：修上述「單一 LINESTRING 管線制水閥畫不出來」的 bug。

### 部署後驗收
1. 進入有制水閥標記的計畫 → 回到大地圖總覽（`showProjectPipelines`）→
   稍等一下（背景載入中）應該會看到紅色圓點標記逐漸出現在對應管線上。
2. 滑鼠移到紅點上顯示「🔧 制水閥」提示；點擊顯示所屬工程名稱的小視窗。
3. 切換到別的計畫、或點進某個工程再返回 → 舊的紅點不該殘留、也不會疊加變多。

   應該要能正常看到紅十字了（這是本次順便修的 bug）。

## pipe-map-worker/src/index.js

### 1. 新增「寫入動作權限驗證」中介層（最重要的修正）
原本前端雖然會在寫入時附上 `userToken`，但 **Worker 從頭到尾沒有讀過它**——
全部 85 個 action（包含 `deletePipeline`、`setUserRole`）任何人都能用一支 curl 直接呼叫，
搭配 CORS `*`，等於整個資料庫對全世界開放寫入。

現在：
- 寫入動作（`save/update/delete/add/clear/upload/set/import/init/batch` 開頭）
  必須附有效的 Google ID Token。
- Token 以 **Google JWKS 公鑰驗 RS256 簽章**，並檢查 `aud`（client id）、`iss`、`exp`，
  不是只解 payload 看看而已；JWKS 有模組層快取（1 小時），不會每次請求都打 Google。
- 驗過身分後查 `users` 表的角色：一般寫入需 contractor(2) 以上；
  `setUserRole / updateUserRole / deleteUser` 需 admin。
- 驗證失敗回 `{ authError: true }`（HTTP 401），前端既有的 reauth 流程可直接銜接。
- 讀取（`get*`）維持免登入，訪客瀏覽體驗不變。
- `verifyUser / registerUser / syncUser` 為登入握手，維持免 token。
- 驗證通過後把 `params._authEmail / _authRole` 帶給後續 handler，日後要做操作紀錄可直接用。

### 2. `deletePipeline` 改用 `db.batch()`
原本 10 條 DELETE 逐條 `await`（10 次網路來回，中途失敗會留半刪狀態）。
改為單一 batch：**1 次來回、單一交易（全刪或全不刪）**，並補齊
`accounting / accounting_by_code / contract_amount / pipeline_codes / method_prices / sticky_notes`
六個原本漏刪的表。

### 3. `updatePipeline` 修「改工程編號 → 全部子資料變孤兒」的重大 bug
編輯視窗允許修改工程編號，但原本只 UPDATE `pipelines` 一張表——
一旦改了 ID，該工程的段落、小段、照片、甘特圖、里程碑、路證……全部斷鏈消失。
現在改 ID 時用 `db.batch()` 原子性地同步更新 15 張子表的 `pipeline_id`。

### 4. 新增 `updateGanttOrder` action
甘特圖視窗的「拖拉排序」一直在呼叫這個 action，但後端根本沒有這個 case，
排序**從來沒有存進資料庫過**（只會跳「排序儲存失敗」）。現已實作（batch 更新 `sort_order`）。

### 5. `saveSegment` 小段插入改 batch
原本一個 100m 大段要跑 10 次 `await ... run()`；改成一次 `db.batch()`，
大段越長效益越明顯，且不會插到一半失敗。

### 6. `getPipelines` 的編號查詢改 JOIN 過濾
原本 `GROUP_CONCAT` 掃**全部**工程的 `pipeline_codes` 再在 JS 端挑；
改為 JOIN `pipelines` 只撈該計畫的，資料量成長後不會越查越慢。

### 7. 移除死掉的 `calcNumSegs()`
它還帶著已被廢棄的「尾段 <5m 併入前一段」邏輯，但實際上已無任何地方呼叫。
留著只會讓人日後誤用，重新引入上一版修掉的「小段數量前後端不一致」問題。
